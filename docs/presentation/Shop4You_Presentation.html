Problem Statement

"Shop4You, a UK-based online retailer, needs an AI assistant that can handle both customer inquiries and internal employee questions across multiple departments — with smart escalation for complex complaints."

😤

Challenge 1

8 departments with different contexts, tones, and audiences

🔀

Challenge 2

Queries sometimes span multiple departments needing handoffs

🔥

Challenge 3

Angry customers need escalation, not generic FAQ answers

Goal: Build a production-ready multi-agent system using LangGraph + CrewAI + RAG + GPT-4o

What Shop4You Does

1
Authenticates — identifies user as Customer or Employee (with loyalty tier / department)
2
Classifies — routes to 1 of 8 departments + detects sentiment + severity (uses last 3 turns)
3
Routes smartly — only HIGH-severity negative → CrewAI escalation. Mild/medium → RAG
4
Retrieves relevant FAQ snippets from ChromaDB knowledge base (200 vectors, 8 departments)
5
Augments with tool calls — order lookup, product search, loyalty points, department listing
6
Generates a grounded answer using GPT-4o with retrieved context + tool results + user profile
7
Reflects on its own answer (quality 1–10). If score < 7, regenerates up to 3 retries
8
Transfers across departments — detects cross-dept queries via [SUGGEST_TRANSFER] tags
9
Remembers conversation per user via SQLite-backed persistent memory
10
Follows up — asks "Anything else?" and handles farewell responses gracefully

Smart Routing & Classification

Classification Output (Structured)

class QueryClassification(BaseModel):
    department: str   # 1 of 8 keys + "unknown"
    sentiment: Literal["positive",
                       "neutral", "negative"]
    severity: Literal["low", "medium", "high"]
    reasoning: str    # why this classification

Routing Decision Tree

if dept == "unknown":
    → escalate  # can't match any dept
elif sentiment == "negative"
     and severity == "high":
    → escalate  # very angry / threats
else:
    → department_rag_agent  # everything else

Severity-Based Escalation

Severity Behaviour Example
Low RAG handles normally "Could be better"
Medium RAG with empathy "Disappointed with quality"
High → CrewAI escalation "I will sue!", Threats

Why not escalate all negatives?

Mild complaints like "a bit disappointed" are common and answerable — sending them to a 3-agent crew wastes resources and gives a worse user experience.

Retrieval Flow

1 Query embedded via text-embedding-3-small
2 ChromaDB similarity search (cosine, top-k=3)
3 Filter by department metadata (score ≥ 0.2)
4 Fallback: 0 dept results → global search + NOTE tag
5 Context injected into GPT-4o RAG prompt

💡 Smart Fallback

When no results in the target department, searches all depts and prepends [NOTE: No results in X...] — prompting the agent to suggest a department transfer.

Reflection & Self-Correction

Quality Assessment Criteria

1️⃣ Relevance — Does it address the query?

2️⃣ Groundedness — Based on context only? (No hallucination)

3️⃣ Completeness — Fully answers the question?

4️⃣ Tone — Appropriate for the situation?

5️⃣ Actionability — Clear next steps for user?

class QualityCheck(BaseModel):
    is_relevant: bool
    is_grounded: bool
    is_complete: bool
    quality_score: int  # 1-10
    feedback: str

Reflection Loop Flow

department_rag_agent │ generates answer ▼ ┌──────────────┐ │ reflect │ │ │ │ GPT-4o runs │ │ quality check│ │ │ │ score ≥ 7? │ │ ┌─────────┐ │ │ │ YES │──┼──▶ END ✅ │ └─────────┘ │ │ ┌─────────┐ │ │ │ NO │──┼──▶ regenerate │ └─────────┘ │ with feedback │ │ │ │ retries < 3? │ │ │ YES → loop ◄─┼────────┘ │ NO → accept │ └──────────────┘

Cross-Department Handoffs

How Transfer Detection Works

1
SUGGEST_TRANSFER tag — Agent embeds [SUGGEST_TRANSFER: dept_key] in response
2
Tag parsing — Orchestrator strips tag, populates suggested_transfer
3
Fallback detection — If no tag but response says "reach out to Shipping", best-match scoring finds the dept
4
User confirms — Streamlit shows transfer button; API uses /transfer endpoint
5
Re-run — run_query_for_department() skips classification, goes straight to RAG

Transfer Keyword Detection

connect me · transfer me · speak to · talk to · chat with · put me through · redirect me · route me · switch me · can i speak · can i talk · can i chat · let me talk · i need to speak · i want to speak · i want to talk · connect to · transfer to · pass me to · pass me · customer service · speak with · talk with

22 phrases — when detected, agent introduces itself as the department and asks how to help

Conversation-Aware Context

The classifier feeds the last 3 conversation turns so that "pass me to customer service" after a shipping conversation → routes to shipping_delivery, not unknown.

Memory & User Management

Persistent Memory

• SqliteSaver from LangGraph — file-backed checkpointer

• Each user gets isolated thread: user_{email}

• Survives server restarts

• Conversations recalled on return

• Different users never see each other's history

Farewell Handling

25+ farewell phrases detected ("bye", "no thanks", "that's all", etc.)

Friendly sign-off without triggering agent pipeline

Avoids misrouting "no" as a negative sentiment query

Key Design Decisions

🎯 Why severity-based escalation?

Early testing showed all negative queries hit CrewAI — even mild "could be better" ones. Added 3 severity tiers so only HIGH queries escalate. Mild complaints get direct RAG help.

🔀 Why SUGGEST_TRANSFER tags?

Agent suggests a transfer — user confirms. No surprise auto-routing. The machine-readable [SUGGEST_TRANSFER] tag is stripped from the visible response — clean UX.

🤖 Why "You ARE customer service" override?

KB data had 47 instances of "contact our customer service team" — the agent was telling users to contact itself. Fixed in KB data and RAG prompt (defence-in-depth).

💬 Why conversation-aware classification?

Follow-ups like "yes, transfer me" need context. Feeding the last 3 turns prevents "unknown department" misroutes and enables coherent multi-turn conversations.
